1. ¿Qué es JaCoCo y cómo funciona?

Java Code Coverage Library — la herramienta estándar de cobertura en proyectos JVM/Android

JaCoCo (Java Code Coverage) es una biblioteca de código abierto que mide qué porcentaje del código fuente es ejecutado durante una sesión de pruebas. En proyectos Android, es la herramienta de facto integrada con Gradle para análisis de cobertura en tests unitarios JVM.

Mecanismo de funcionamiento

  1. Instrumentación de bytecode. Cuando Gradle compila el código Kotlin, JaCoCo inyecta contadores invisibles dentro del bytecode (.class files). Cada contador está ligado a una línea, rama condicional o método específico del código original.

  2. Ejecución de tests. Al correr ./gradlew testDebugUnitTest, los tests JUnit ejecutan el código de producción. Cada vez que una línea instrumentada es alcanzada, su contador se incrementa.

  3. Volcado de datos. Al finalizar los tests, JaCoCo escribe un archivo binario testDebugUnitTest.exec con todos los valores de los contadores.

  4. Generación del reporte. La tarea jacocoTestReport lee el archivo .exec y genera reportes en formato HTML, CSV y XML. El XML fue la fuente de datos del prototipo.

  5. Cálculo del porcentaje. Para cada elemento (línea, método, clase): cobertura % = cubiertos / (cubiertos + no-cubiertos) × 100.

2. ¿Qué se analizó?

Descripción del proyecto y el alcance del análisis

ProyectoPlanTravel es una aplicación Android nativa construida con Kotlin + Jetpack Compose. Permite a grupos de viajeros coordinar itinerarios, gastos compartidos y comunicación en tiempo real. El backend es Supabase (PostgreSQL + Auth + Realtime).

Nota sobre el alcance de JaCoCo

JaCoCo solo puede medir los tests que corren en la JVM (src/test/). Los tests instrumentados (src/androidTest/) requieren un dispositivo Android o emulador y quedan fuera del alcance de este prototipo. Esto es importante al interpretar los porcentajes de cobertura.

4. Interpretación y diagnóstico

¿Qué dicen realmente estos números?

4.1 Cobertura: 12.1% total — ¿es grave?

Un 12.1% de cobertura de líneas suena alarmante, pero el número total oculta una historia más matizada. El proyecto tiene una arquitectura en capas bien definida, y la cobertura varía radicalmente según la capa:

Capa de dominio — Excelente

domain.settlement (100%), domain.usecase (100%) y domain.model (91%) están completamente cubiertos. Esta es la capa más crítica: contiene la lógica de negocio pura (cálculo de deudas, asignación de gastos) que no depende de Android ni de Supabase. Es exactamente lo que debe estar bien testeado.

Capa de presentación (ViewModels) — Muy buena

Los ViewModels tienen cobertura alta: presentation.group (94.7%), presentation.expense (90.8%), presentation.auth (87.5%). Esto indica que el equipo aplicó correctamente el patrón de inyección de dependencias para poder probar los ViewModels de forma aislada (sin Android).

Capa de UI (pantallas Compose) — 0% — Esperado y estructural

ui.screens tiene 2,018 líneas con 0% de cobertura. Esto no es negligencia: las pantallas Compose (@Composable) solo pueden probarse con un emulador o dispositivo Android real, lo que requiere tests instrumentados que JaCoCo no captura en este contexto. Es una limitación inherente al stack tecnológico, no un error del equipo. Este paquete pesa tanto (2,018 de 3,007 líneas no cubiertas = 67% del déficit) que arrastra el porcentaje global hacia abajo artificialmente.

Capa de datos — 0% — Justificable pero mejorable

data.repository y data.auth tienen 0% porque sus implementaciones llaman directamente a Supabase por red. Probarlos en unit tests requiere mocks de los clientes HTTP, lo cual añade complejidad. Sería mejorable con tests de integración o mocking más exhaustivo.

4.2 Cobertura "real" excluyendo UI no-testeable

Si excluimos ui.screens y ui.components (que requieren dispositivo) y data.* (que requieren red), la cobertura del código efectivamente testeable por JVM es aproximadamente 72–85%. Eso es un indicador de calidad alto.

4.3 Densidad de defectos: 2.51 defectos/KLOC

Contexto de la industria

Según estudios de ingeniería de software (McConnell, Code Complete; NIST): proyectos sin proceso formal suelen tener 15–50 defectos/KLOC antes de pruebas. Con buenas prácticas, el rango post-testing baja a 1–5 defectos/KLOC. El resultado de 2.51 se ubica en el rango bueno para un proyecto en desarrollo activo.

Un dato notable: 10 de los 15 commits de defecto corresponden a ciclos sistemáticos de diagnóstico y corrección relacionados con issues numerados (#20 al #25). Esto sugiere que los bugs no surgieron de forma caótica sino que fueron descubiertos y resueltos de manera organizada durante fases de QA planificadas — lo cual es una señal positiva del proceso, no negativa.

5. ¿Cómo mejorar las métricas?

Acciones concretas ordenadas por impacto

Para aumentar la cobertura de código

  1. Agregar tests para ui.utils (actualmente 62.9%). Funciones utilitarias puras como formatExpiry son triviales de testear y aumentarían la cobertura de ese paquete al 100% con pocos tests adicionales.

  2. Testear los DestinationViewModel e ItineraryViewModel (actualmente 0%). Los otros ViewModels están cubiertos; estos dos se quedaron sin tests. Siguiendo el patrón ya existente en ExpenseViewModelTest se pueden agregar en pocas horas.

  3. Agregar tests de UI con Compose Testing. La biblioteca androidx.compose.ui:ui-test-junit4 (ya en el proyecto) permite escribir tests instrumentados que corren en emulador y cubren las pantallas Compose. Esto requiere configurar un emulador en CI pero es el paso de mayor impacto potencial.

  4. Mockear la capa de datos con FakeTravelRepository. El proyecto ya tiene FakeTravelRepository y FakeAuthRepository creados. Expandir su uso para cubrir más casos de borde en los tests de ViewModel aumentaría la cobertura de ramas (branch coverage) del 13.8% actual.

Para reducir la densidad de defectos

  1. Adoptar TDD (Test-Driven Development) para nuevas funcionalidades. Escribir el test antes que el código obliga a pensar en los casos de borde antes de que se conviertan en bugs. Las funcionalidades del dominio ya son las menos defectuosas precisamente porque tienen mejor cobertura.

  2. Revisión de código (code review) antes de mergear. Un segundo par de ojos antes de integrar código detecta errores lógicos que los tests no capturan (mal manejo de estados, race conditions en coroutines, etc.).

  3. Análisis estático con Detekt o ktlint. Herramientas como Detekt analizan el código Kotlin sin ejecutarlo y detectan patrones propensos a errores (null safety, complejidad ciclomática alta, code smells). Se integran como tarea Gradle.

  4. Definir criterio de aceptación con tests antes de cerrar un issue. Actualmente los bugs se detectan en diagnóstico post-implementación. Si cada issue de funcionalidad incluye tests de aceptación, los defectos se detectan antes.

6. Observaciones adicionales

Métricas secundarias y patrones observados

Ratio test/producción

El proyecto tiene 1,403 LOC de test contra 5,988 LOC de producción, lo que da un ratio de 23.4%. En proyectos maduros se apunta a 50–100% (paridad o mayor). Para una app Android en etapa MVP con foco en features, 23.4% es aceptable pero con margen de mejora.

Distribución del código por capa

El paquete ui.screens concentra 2,018 de las 3,007 líneas no cubiertas (67%). Esto refuerza que el problema de cobertura no es falta de cultura de testing, sino la naturaleza de las pantallas Compose que requieren hardware Android para ser probadas.

Patrón de defectos: diagnóstico sistemático

10 de los 15 commits de defecto provienen de sesiones de diagnóstico planificadas (issues #20–#25), no de regresiones descubiertas en producción. Esto indica que el equipo realiza QA iterativo por feature antes de marcar issues como completados — un proceso de calidad más maduro de lo que el número de defectos podría sugerir.

Velocidad de resolución (MTTR aproximado)

Aunque el prototipo no calculó MTTR formalmente (requiere timestamps de apertura de issue vs. cierre), el historial muestra que los commits de corrección siguen inmediatamente a los commits de diagnóstico, sugiriendo un tiempo medio de resolución menor a 24 horas para defectos detectados.
